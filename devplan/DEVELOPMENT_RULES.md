# QuickTranslate 开发规范

## 1. 文档目的

本规范用于约束 QuickTranslate 项目的后续开发方式、任务执行节奏、架构选型与代码组织原则。

后续所有开发工作默认遵循本文件，若有新增规则，再继续补充。

## 2. 协作规则

### 2.1 任务逐个完成

- 所有开发任务必须逐个推进，不并行展开多个实现任务。
- 每完成一个任务后，先进行本地编译验证。
- 只有在当前任务编译通过后，才能进入向你汇报和确认的阶段。
- 在你明确确认当前任务通过之前，不进入下一个任务的开发。

### 2.2 每个任务的标准执行流程

每个任务都遵循以下流程：

1. 明确当前任务目标与范围
2. 实现当前任务
3. 执行编译或必要校验，确保当前代码可通过
4. 汇报当前任务完成情况、改动点与验证结果
5. 等待你确认
6. 你确认后，再开始下一任务

### 2.3 禁止跳步

- 不允许在未完成当前任务确认前，提前实现下一个任务内容。
- 若当前任务中发现额外问题，优先控制在当前任务范围内处理。
- 若问题超出当前任务范围，需要先说明，再等待你决定是否插入处理。

## 3. 架构规范

### 3.1 页面交互架构

- 所有交互页面采用 `MVI` 架构开发。
- 页面状态必须围绕单一状态源组织。
- 页面层统一按以下思路设计：
  - `Intent`：用户意图与页面事件输入
  - `State`：页面状态
  - `Effect`：一次性事件，例如 Toast、导航、错误提示
  - `Reducer` 或等价状态收敛机制：负责状态演进

### 3.2 状态管理原则

- 页面 UI 只消费状态，不直接拼接业务逻辑。
- 业务逻辑尽量放入 ViewModel / Intent Handler / UseCase 中。
- 避免在页面中散落临时状态和多处状态源。
- 同一页面的状态变更路径必须清晰、可追踪、可测试。

## 4. 模块化规范

### 4.1 组件化要求

- 项目采用组件化架构开发。
- 不同组件必须拆分到不同模块，不允许所有功能堆积在单一模块中。

### 4.2 模块拆分原则

建议至少按职责拆分为以下方向：

- `app`
  - 应用入口、导航装配、全局初始化
- `core`
  - 通用基础能力，例如网络、数据库、公共工具、设计系统
- `feature`
  - 具体业务功能模块，例如首页、学习页、转写页
- `domain`
  - 业务实体、UseCase、仓库接口
- `data`
  - 数据实现层，例如本地存储、网络实现、模型数据源

说明：

- 后续具体模块名可根据实际工程调整，但必须遵守“按职责拆分、避免耦合堆叠”的原则。
- feature 模块之间禁止随意直接依赖，优先通过 domain 或公共接口解耦。

### 4.3 模块边界原则

- UI 相关代码放在 feature 模块
- 数据库存储实现放在 data/local 或等价模块
- 网络请求实现放在 data/remote 或等价模块
- 通用依赖注入、日志、工具类放在 core 模块
- 不允许把与当前模块无关的实现混入模块内部

### 4.4 模块文档要求

- 每新增一个模块，必须在该模块根目录创建模块介绍文档。
- 模块介绍文档文件名统一为 `MODULE.md`。
- 该文档用于说明该模块的职责、目录结构、对外暴露能力、依赖关系与后续维护注意事项。
- 修改任何模块中的实现时，如果涉及职责变化、结构变化、接口变化或重要实现变化，必须同步更新该模块根目录下的 `MODULE.md`。
- 模块代码与模块文档必须保持一致，禁止只改代码不改说明。

### 4.5 项目索引文档要求

- 项目根目录必须维护一份项目介绍文档，作为全项目说明与模块文档索引入口。
- 项目介绍文档文件名统一为 `PROJECT_OVERVIEW.md`。
- `PROJECT_OVERVIEW.md` 中必须记录每个模块对应的 `MODULE.md` 路径，方便快速检索。
- 每新增模块时，除了新增模块根目录下的 `MODULE.md`，还必须同步更新根目录的 `PROJECT_OVERVIEW.md`。
- 每删除、重命名或重构模块时，也必须同步更新 `PROJECT_OVERVIEW.md` 中的索引信息。

## 5. 技术选型规范

### 5.1 指定开源库

项目后续开发优先采用以下技术栈：

- 依赖注入：`Koin`
- 本地数据库：`Room`
- 网络基础库：`OkHttp`
- 网络请求封装：`Retrofit`

### 5.2 使用原则

- 新功能优先基于以上库集成，不重复造轮子。
- 若确需引入额外三方库，需优先确认其必要性、维护情况与项目价值。
- 同类能力避免重复引入多套方案，例如已有 `Room` 时不再并行引入另一套本地数据库框架。

### 5.3 依赖注入使用规范（防止滥用）

- 依赖注入只用于“跨模块边界”或“需要替换实现”的对象：
  - Repository 接口与实现
  - 数据库 / 网络客户端 / 队列引擎 / 播放器等基础设施
  - ViewModel、跨页面共享协调器
- 模块内部、仅被单一调用链使用的轻量对象，优先在上层直接 `new` 组合，不单独注册到 DI 容器。
- 不为“纯转发”逻辑强制创建 UseCase（仅一层调用仓库且无业务规则时）：
  - 允许直接在 ViewModel/Coordinator 使用对应 Repository。
- 仅当出现以下场景时再新增独立 UseCase：
  - 存在明确业务规则编排（校验、事务边界、重试、状态流转）
  - 一个能力被多个调用方复用
  - 需要独立单测和长期演进
- 禁止“为抽象而抽象”：
  - 单个模块内不应出现大量仅被一次使用、无状态、无替换需求的 DI `factory`。
  - 不应为了“看起来分层完整”而堆叠薄封装类。

依赖注入评审检查项（PR 必查）：

1. 新增 DI 注册是否存在明确边界价值（可替换/跨模块/共享生命周期）？
2. 新增 UseCase 是否包含真实业务逻辑，而不是纯转发？
3. 该对象是否只在单链路单点使用，若是应改为本地组合？
4. 变更后是否减少了构造参数噪音与容器复杂度？

## 6. 代码组织规范

### 6.1 类长度控制

- 每个类应保持职责单一。
- 若某个类代码过长、逻辑过于复杂或承担了多个职责，必须主动拆分。

### 6.2 拆分类的优先方式

优先使用组合，而不是继续膨胀单个大类。

推荐拆分方式包括：

- 将状态定义拆分为独立文件
- 将复杂 UI 拆为多个子组件
- 将业务逻辑拆为 UseCase / Coordinator / Handler
- 将数据转换拆为 Mapper
- 将播放器、循环、字幕匹配等能力拆为独立组件

### 6.3 禁止事项

- 不编写超长“上帝类”
- 不把页面、状态、业务逻辑、数据访问全部塞进一个文件
- 不在 ViewModel 中堆叠所有逻辑而不抽取可复用组件

## 7. 代码质量要求

### 7.1 基本要求

- 所有提交代码必须保证至少可编译通过
- 新增代码应尽量保持可读、可维护
- 命名应清晰表达职责，不使用含糊缩写

### 7.2 可维护性要求

- 优先写易于理解的实现，而不是过度炫技
- 复杂逻辑需要有清晰结构，必要时增加简洁注释
- 对未来可能扩展的能力，优先预留合理接口，而不是提前过度设计

## 8. 默认执行约定

后续开发默认按以下方式执行：

1. 从 `MVP_DEV_TASKS.md` 中选定一个最小任务
2. 严格按本规范完成实现
3. 编译验证通过后向你汇报
4. 等你确认后再进入下一任务
