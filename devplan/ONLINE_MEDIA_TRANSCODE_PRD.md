# 在线链接转码入口 PRD（音视频链接 -> 本地音频 + 字幕）

## 1. 文档信息

- 文档名称：在线链接转码入口 PRD
- 目标版本：V1（第一期落地）
- 目标平台：Android
- 关联文档：
  - MVP PRD：`/devplan/MVP_PRD.md`
  - 转码队列方案：`/devplan/TRANSCODE_QUEUE_PIPELINE_PLAN.md`
  - Whisper 接入说明：`/devplan/WHISPER_INTEGRATION.md`
  - 开发任务拆解：`/devplan/ONLINE_MEDIA_TRANSCODE_DEV_TASKS.md`

## 2. 背景与问题

当前 QuickTranslate 的导入入口仅支持系统文件选择器导入本地音视频文件；用户常见素材来源为网页/播客/云盘/课程链接，导入路径成本高（需要先下载到本地再手动选择）。

新增“链接导入 + 转码”入口，目标是：

- 用户只需粘贴一个音/视频链接，即可在 App 内完成下载、生成字幕，并进入学习页。
- 产物必须落本地：可离线播放的媒体文件（至少音频） + 可复用的字幕数据（SRT 解析后的片段）。

## 3. 目标与非目标

### 3.1 目标（V1）

1. 支持用户输入/粘贴 `http/https` 音视频直链，创建项目并进入“转码队列”。
2. 在后台串行执行（复用现有队列引擎约束：同一时刻仅 1 个任务运行）。
3. 下载完成后生成可离线使用的本地媒体文件，并启动本地转写生成字幕。
4. 全流程可观测：排队/下载中/转写中/成功/失败，且失败可重试。

### 3.2 目标（V2：第三方平台抓取/解析）

在 V1 直链能力基础上，支持用户输入“平台页面链接”，App 端自动解析出可下载的媒体资源（音频或视频），并复用同一套“下载 -> 本地媒体 -> 转写 -> 学习”闭环。

V2 的关键目标：

1. 支持“平台链接 -> 可下载直链/流 -> 本地文件”的解析能力（含必要的 Header/Cookie/Referer 注入）。
2. 解析结果可回落为“可播放清晰度列表”（如多码率/多音轨），用户可选择后再入队下载。
3. 解析失败时给出明确原因（需要登录/区域限制/DRM/站点不支持），并提供兜底方案（浏览器打开、复制直链提示等）。

### 3.3 非目标（V1 不做）

- m3u8/HLS 等流媒体切片下载与合并（若系统层可直接下载为单文件则不限制，但不专门适配）。
- 断点续传与跨进程强保活（进程被杀后的恢复按现有“首页加载后 restoreAndSignal()”语义处理）。
- 多任务并行下载/转码。

### 3.4 非目标（V2 仍不做）

- DRM 内容解密、付费内容抓取、绕过平台访问控制。
- 需要强登录态/复杂交互的站点适配（可先做“登录后在 WebView 内解析 Cookie”但不承诺覆盖所有场景）。
- “全平台通用”解析承诺：V2 只按“站点白名单”逐个适配与验收。

## 4. 用户故事

- 作为学习者，我希望把一个播客/课程的音频链接粘贴到 App 中，自动下载并生成字幕，这样我无需切换到外部下载器。
- 作为学习者，我希望导入后可以离线学习，这样我在无网环境也能听/看字幕。
- 作为学习者，我希望看到当前任务处于“下载/转写/排队”等状态，并能在失败后重试。

## 5. 核心流程（端到端）

1. 用户在首页点击“导入链接”（或从系统分享进入链接导入）。
2. 输入/粘贴链接，点击“开始导入”。
3. App 立即创建项目记录（状态为“待转码”），并入队一个“链接转码任务”。
4. 转码队列串行执行任务：
   - 下载远程媒体到项目目录
   - 生成/准备本地音频（至少保证可用于离线播放与转写输入）
   - 调用现有转写 pipeline 生成字幕并落库
5. 用户进入学习页：
   - 若任务仍进行中：展示下载/转写状态与进度（可返回，后台继续）
   - 成功：正常播放 + 字幕联动
   - 失败：展示错误信息 + 重试入口

## 6. 功能需求

### 6.1 新入口（导入链接）

- 首页新增入口：
  - 主按钮旁新增“导入链接”入口（或在导入弹窗中提供“文件/链接”切换）
- 支持系统分享：
  - 支持从浏览器/聊天工具“分享链接到 QuickTranslate”，直接打开链接导入页并填充 URL
- 支持剪贴板快捷：
  - 进入链接导入页时，若剪贴板存在 `http/https` URL，提供“一键粘贴”提示

### 6.2 链接校验与预检查

- 仅接受 `http/https`，拒绝 `file://`、`content://`、`javascript:` 等非预期 scheme。
- 基础校验：
  - URL 格式校验
  - 可选：域名黑名单（本期可不做）
- 预检查（可选，V1 建议做最小化）：
  - 使用 `HEAD` 或 `GET range` 获取 `Content-Type`、`Content-Length`（若服务端不支持则降级）
  - 仅允许 `audio/*`、`video/*`，或 URL 后缀命中常见媒体格式（作为弱规则）

### 6.2.1 平台链接识别（V2）

当 URL 不满足“直链特征”时，进入“平台链接解析”分支：

- 直链判定（建议规则）：
  - `Content-Type` 为 `audio/*` 或 `video/*`（预检查拿到时）
  - 或 URL 后缀命中常见媒体扩展名（mp3/m4a/aac/mp4/mkv/webm 等）
- 平台链接判定：
  - 不满足直链判定，且 URL host 命中“站点白名单”

站点白名单（V2 需产品明确，默认 TBD）：

- **首批站点（确定）**：
  - Bilibili（B 站）：`www.bilibili.com` / `m.bilibili.com` / `b23.tv`
  - 抖音：`www.douyin.com` / `v.douyin.com` / `www.iesdouyin.com`

说明：

- V2 只对以上 host 的“公开视频页面链接”提供解析与验收。
- 需要登录/付费/DRM/区域限制的内容不承诺可解析（按失败分类提示）。

### 6.3 下载与本地缓存

- 下载实现建议基于 `OkHttp`（项目技术选型已指定）。
- 下载存储位置：
  - App 私有目录（如 `filesDir`/`cacheDir` 下按 `projectId` 分目录）
  - 文件命名需稳定：便于恢复与清理（例如 `source.media`、`audio.wav`、`audio.m4a`）
- 下载状态：
  - `queued`（排队中）
  - `downloading`（含进度百分比/已下载字节）
  - `download_failed`（错误码/错误信息）
  - `download_succeeded`
- 取消与清理：
  - 用户删除项目时必须同时取消活跃任务并删除已下载文件与派生产物
  - 任务失败后允许“重试”，重试前可选择是否复用已下载文件（默认复用，除非文件不完整）

### 6.4 本地音频生成策略（“本地音频”定义）

V1 对“本地音频”的最低定义：

- 任务完成后，项目必须具备一个可离线访问的本地媒体文件路径，可用于学习页播放；
- 转写输入必须为本地文件路径（避免直接对远程 URL 做转写输入，提高稳定性）。

推荐实现策略（按优先级）：

1. **若输入是音频直链**：下载后直接作为项目媒体文件（音频）。
2. **若输入是视频直链**：
   - V1 允许直接下载并以视频作为项目媒体文件（学习页可播放视频/或仅播放音轨）
   - 同时在转写阶段从本地视频文件解码音轨生成 WAV 作为转写输入（可复用现有 `SessionMediaPrepareStage` 思路）
3. 音频“长期保存格式”：
   - 若后续需要节省空间，V2 再实现“抽取并编码为 m4a/aac”的音频持久化；V1 不强制要求（避免引入 ffmpeg 等重依赖）。

### 6.5 字幕生成与落库

- 复用现有字幕生成能力：
  - 转写服务：`mock` / `whisper-cli(real)` 现有模式保持不变
  - 字幕解析：使用现有 SRT 输出与解析逻辑
- 字幕数据落库后，学习页必须可无网直接加载并播放联动。

### 6.6 任务队列与状态联动

要求与现有队列策略保持一致：

- 串行执行：同一时刻最多 1 个运行中任务
- 支持提权：用户进入学习页时，对应项目任务提升到队首（不抢占正在运行的任务）
- 后台继续：离开学习页后继续执行
- 恢复：进程重启后，首页加载完成触发恢复并继续 drain 队列

任务类型建议扩展（V1 可先合并为一个任务，V2 再拆分）：

- `DOWNLOAD_AND_TRANSCRIBE`：单任务内串行完成下载 + 转写
- 或拆分为：
  - `DOWNLOAD`
  - `TRANSCRIBE`

### 6.7 UI 展示（最小集）

- 首页项目卡片：
  - 显示字幕状态（与现有一致）
  - 若为链接项目，额外显示“来源：链接”标识（可选）
- 转码看板：
  - 新增下载阶段状态文案与进度
- 学习页：
  - 未完成时：展示“下载中/转写中/排队中”与可返回提示
  - 失败时：展示失败原因 + 重试按钮

## 7. 数据模型与存储（建议稿）

### 7.1 Project 扩展字段（建议）

为避免复用 `mediaUri` 承载过多语义，建议新增字段（domain/data 层落地时再评估迁移）：

- `sourceType: LOCAL_FILE | REMOTE_URL`
- `sourceUrl: String?`（原始链接）
- `localMediaPath: String?`（下载后的本地媒体文件路径）
- `localAudioPath: String?`（可选：若做音频抽取的长期保存）
- `downloadStatus / downloadProgress`（可选：若不想扩 Project，可仅放到任务表）

V1 也可采取“最小改动策略”：

- `Project.mediaUri` 存下载后的本地路径
- 原始 URL 存在 `ProjectTranscodeTask.mediaUri`（或单独新增字段）

### 7.2 转码任务表扩展（建议）

在现有 `project_transcode_task` 基础上增加下载阶段的可观测字段（按需）：

- `stage: DOWNLOADING | PREPARING | TRANSCRIBING | PERSISTING`（或直接复用 status + error）
- `progress: Int`（0-100；可把“下载进度/转写进度”统一到一个维度）
- `bytesDownloaded/bytesTotal`（可选，用于更准确的进度与诊断）

## 8. 约束、风险与合规

- 链接兼容性风险：不同站点可能不返回 `Content-Length`、不支持 Range、或需要 Cookie/Referer；V1 明确仅支持“无需鉴权的直链”。
- DRM/版权：不提供平台抓取能力；用户需确保拥有下载/使用权。导入页需增加轻提示文案。
- 存储占用：视频下载体积大；V1 需要在导入前展示预估大小（若可获取）并提醒空间占用；提供项目删除即清理的确定性行为。
- 网络与电量：下载/转写耗时，需在 UI 中明确“可后台执行、可能耗电”。

## 9. 验收标准（V1）

1. 输入一个可访问的音频直链（如 mp3/m4a），可完成下载并生成字幕，进入学习页可离线播放 + 字幕联动。
2. 输入一个可访问的视频直链（如 mp4），可完成下载并生成字幕，进入学习页可播放（视频或音轨）+ 字幕联动。
3. 任务在队列中串行执行，且学习页进入会提升该项目任务优先级（不抢占运行中任务）。
4. 失败场景可复现并可重试（至少覆盖：网络失败、链接不可达、无音轨、转写失败）。
5. 删除项目会清理本地下载文件与字幕数据，且不会遗留“RUNNING 永久卡死”的任务（按现有 restore 语义可恢复）。

## 9.1 验收标准（V2：第三方平台抓取/解析）

1. 输入一个“平台页面链接”（站点在白名单内），可解析出至少 1 个可下载条目，并能完成下载 + 转写 + 学习闭环。
2. 若平台需要登录/DRM/区域限制，App 能明确提示原因，并不会进入“无限重试/卡死”状态。
3. 解析出的下载条目若依赖 Header（Cookie/Referer），下载阶段必须正确携带，且失败可诊断（日志/错误信息）。
4. 站点不在白名单时，明确提示“不支持该站点”，并提供“在浏览器打开”入口。

### 6.8 平台抓取/解析能力（V2）

#### 6.8.1 解析输出

平台解析的输出建议统一为“可下载条目列表”，每个条目包含：

- `title`（可选，用于默认项目名）
- `duration`（可选，用于导入前确认）
- `items[]`：
  - `qualityLabel`（如 360p/720p/仅音频）
  - `mimeType` / `container`（mp4/webm/m4a）
  - `sizeBytes`（可选，拿不到可为空）
  - `downloadUrl`（可能是一次性 URL，可能需要 Header）
  - `headers`（可选：Cookie/Referer/User-Agent）

#### 6.8.2 解析方式（技术方案约束）

V2 允许以下方案之一（需研发评估后定稿）：

1. **站点官方 API**（若有公开接口、且合规可用）：稳定但覆盖有限，可能需要 key。
2. **内置解析器（白名单站点逐个适配）**：通过抓 HTML/JSON、站点接口等提取媒体直链；工作量与维护成本高。
3. **集成通用解析引擎（如引入第三方开源提取器/CLI）**：覆盖广，但体积、更新、合规与安全风险更高，需要严格白名单与版本治理。

V2 的最小可交付建议：先选 1 个站点完成“端到端”，再扩展站点白名单。

#### 6.8.2.1 站点实现顺序建议（V2）

- **V2-1：优先 B 站**（可控性相对更高，且“长视频 + 字幕学习”匹配度更强）
- **V2-2：抖音**（短视频更碎片化，且站点侧风控更强，需预期更高维护成本）

#### 6.8.2.2 Bilibili（B 站）解析要点（V2）

- 支持链接形态：
  - `https://www.bilibili.com/video/...`
  - `https://b23.tv/...`（需要先展开重定向）
- 解析策略（建议）：
  - 先通过直链预检查判断是否已是可下载媒体；否则进入平台解析
  - 允许用户在内置 WebView 打开页面并登录（若用户愿意），获取 Cookie 作为后续解析请求 Header（不做绕过）
  - 输出可选清晰度列表（至少“仅音频/默认清晰度”二选一，拿不到时可降级为单选）
- 失败分类重点：
  - `LOGIN_REQUIRED`：需要登录或年龄/等级限制
  - `REGION_RESTRICTED`：番剧/版权区限制
  - `DRM_PROTECTED`：DRM 或加密流（不支持）

#### 6.8.2.3 抖音解析要点（V2）

- 支持链接形态：
  - `https://www.douyin.com/video/...`
  - `https://v.douyin.com/...`（短链，需要先展开重定向）
- 解析策略（建议）：
  - 同样允许 WebView 登录以获取 Cookie（若用户愿意）
  - 由于站点侧参数/签名机制变化频繁，V2 明确“仅对白名单内、公开可访问内容尽力解析”，失败按分类提示
- 失败分类重点：
  - `EXTRACT_FAILED`：站点结构变更/签名参数不匹配
  - `LOGIN_REQUIRED`：需要登录或内容仅粉丝可见
  - `REGION_RESTRICTED`：内容不可访问

#### 6.8.3 失败分类与用户提示

- `UNSUPPORTED_SITE`：站点不在白名单
- `LOGIN_REQUIRED`：需要登录态
- `REGION_RESTRICTED`：区域限制/版权限制
- `DRM_PROTECTED`：DRM 或加密流
- `EXTRACT_FAILED`：解析失败（站点结构变更等）

UI 必须能展示失败分类与简短可操作提示。
